## 4. 原利润表费用列标题与当前回款源字段的历史错位

这是本次最需要回查的异常之一。

结论：

- 原利润表 [01-TK文件/2026年菲律宾2月tiktok订单利润表（1.1%）.xlsx](01-TK文件/2026年菲律宾2月tiktok订单利润表（1.1%29.xlsx) 中，Tiktok订单完成表 的部分费用列标题，与当前回款源表的真实字段含义存在历史错位。
- 当前脚本在生成程序版订单完成表时，按“源回款表真实字段语义”写入，不再沿用旧错位口径。

用于确认错位的样例订单：

- 订单号：582836594225481048
- 店铺：2月份-阳玲母婴

该订单在回款源表中的真实非零字段为：

- Transaction fee = -2.15
- TikTok Shop commission fee = -5.7
- Affiliate Shop Ads commission = -8.91
- Bonus cashback service fee = -1.78
- Shipping service fee = -4.9
- Order processing fee = -3

而原利润表中同一订单对应的非零落点表现为：

- Affiliate commission deposit = -8.91
- Pre-order service fee = -3
- LIVE Specials service fee = -1.78
- Campaign resource fee = -4.9

据此可以确认以下历史错位关系：

| 当前回款源真实字段 | 原利润表中实际落到的列标题 |
| --- | --- |
| Affiliate Shop Ads commission | Affiliate commission deposit |
| Order processing fee | Pre-order service fee |
| Bonus cashback service fee | LIVE Specials service fee |
| Shipping service fee | Campaign resource fee |

直接后果：

- 原利润表里的 Affiliate Shop Ads commission、Order processing fee、Bonus cashback service fee、Shipping Service Fee 这些列，在样例订单上会显示为 0 或与当前源表语义不一致。
- 如果后续继续沿用原利润表表头直接做聚合，费用分类会继续带错。

这里说的“原表历史错位的费用列口径”，指的不是源回款表字段有问题，而是：

- 原利润表在历史维护过程中，曾把部分真实费用写进了错误的列标题下。
- 因此同一个订单在原利润表里看起来像是“某列有值”，但对照回款源表语义后，会发现这笔值实际属于另一列。

当前已经确认的 4 组错位是：

| 当前回款源真实字段 | 原利润表中实际落到的列标题 |
| --- | --- |
| Affiliate Shop Ads commission | Affiliate commission deposit |
| Order processing fee | Pre-order service fee |
| Bonus cashback service fee | LIVE Specials service fee |
| Shipping Service Fee | Campaign resource fee |

现在脚本会把这部分单独摘到差异核对工作簿的以下工作表里，便于直接观察：

- `3-费用列错位差异`：逐订单明细，能看到程序值、原表同名列值、原表历史落点值、错位判定、改善值。
- `3-费用列错位汇总`：按 `真实字段 + 原表历史落点列 + 错位判定` 汇总，能直接看受影响订单数和金额影响。

其中：

- `错位判定 = 更符合历史错位列` 表示程序值与原表“错误落点列”更一致，而和原表同名列不一致，这类最能说明原表沿用了旧错位口径。
- `历史错位改善值` 越大，表示“如果按历史落点列去理解原表，会比看原表同名列更接近程序值”，也就越能证明这部分差异主要来自历史列错位，而不是程序计算错误。

### 5.2 广告费与空状态调整行现在改为“单表补齐”

当前脚本已经改成“系统自动重建主体层 + 人工调整附加层观察”两层结构。

先由系统自动重建的部分：

- 线上广告费：来自到款明细表中 `Type = GMV Payment for TikTok Ads` 的记录，不再误并入真实订单回款，而是单独生成店铺级广告费调整行。
- 线下广告费：来自营销表可见工作表中的 `线下广告费金额（PHP）`，按店铺维度生成调整行，不拆分到订单。

用于确认的关键样例：

- 到款表订单号：`3561946011828979613`
- 店铺：`2月份-阳玲母婴`
- Type：`GMV Payment for TikTok Ads`
- Total settlement amount：`-26936.96`
- 它在营销表可见工作表中的 `店铺余额充值广告费金额（PHP）` 也能对到 `26936.96`

这说明：

- `GMV Payment for TikTok Ads` 不是完成订单本身，而是店铺级线上广告费回款/扣款调整。
- 这类记录应生成 `Order Status` 为空的广告费调整行，写入 `广告费` 列，而不应参与真实订单匹配。

在系统自动重建之后，广告费导入表的角色才是：把系统仍然无法直接推出的部分，统一压缩为 1 张额外表，不再依赖多份手工表单和临时口径。

这张表同时覆盖两类补充数据：

- 分摊广告费：仅用于“系统没有自动识别到”的额外广告费或特殊广告费场景；不再作为默认主链路。
- 新增调整行：用于补原利润表里 Order Status 为空的人工调整行，例如 Withholding tax、Platform reimbursement、Logistics reimbursement 等。

当前已经确认，原表独有的人工调整层不是只有利润表最后 54 行，还包括前面散落的 20 行同类调整。脚本现在会把它们单独导出到差异核对工作簿中：

- `1-人工调整观察`：逐行观察表，直接区分 `尾部54行补录` 和 `散落补录`
- `1-人工调整汇总`：按 `人工调整层分类 + 备注` 汇总金额影响

目前这 74 行的分布是：

- 尾部 54 行：Withholding tax 28 行、线下广告费/线下退款 18 行、Logistics reimbursement 5 行、Platform reimbursement 2 行、广告费 1 行
- 散落 20 行：Platform reimbursement 15 行、Logistics reimbursement 3 行、Violation fee settlement fee 2 行

设计原则：

- 系统能自动推导的字段，仍然只由 011-系统数据 + 成本表 + 固定税率/汇率生成。
- 线上广告费优先由回款表自动识别；线下广告费优先由营销表可见工作表自动识别。
- 只有系统无法推导的“额外广告费归属规则”和“特殊调整行”才进入广告费导入表。
- 广告费导入表只保留最小人工输入，不再要求按订单逐行手工补利润表。

直接效果：

- 程序生成表中的 广告费 列现在已有系统自动重建来源，不再必须依赖人工导入。
- 原利润表中的部分空状态广告费调整行，已经可以直接由系统数据自动恢复。
- 之后要还原整张利润表，输入集合收敛为：011-系统数据 + 成本表.xlsx + 广告费导入表.xlsx；其中广告费导入表只负责系统无法直接推出的补充部分。

### 5.3 广告费导入表的字段设计

脚本当前支持的导入表工作表名为：`广告费导入表`

当前模板已经做了颜色区分，便于直观看哪些字段要填：

- 黄色：始终必填。
- 橙色：只有 `分摊广告费` 时必填。
- 浅黄色：只有 `分摊广告费` 时按需填写的筛选字段。
- 浅蓝色：只有 `新增调整行` 时按需填写的明细字段。
- 浅绿色：只有 `新增调整行` 时才可能填写的覆盖字段；大多数情况建议留空，让脚本自动重算。

核心字段分两组。

公共字段：

- 启用：1/是/TRUE 表示生效。
- 记录类型：只允许 `分摊广告费` 或 `新增调整行`。
- 月份：用于生成调整行 ID 和回查。
- 店铺：必须与程序生成表里的 店铺 完全一致，例如 `2月份-刘林长3C`。
- 来源单据、来源行号：仅用于留痕和复核。

当 `记录类型 = 分摊广告费` 时：

- 必填：分摊金额(PHP)、分摊方式、店铺。
- 可选筛选：筛选_是否到款、筛选_Order Status、筛选_Seller SKU、筛选_开始时间、筛选_结束时间。
- 支持分摊方式：按订单金额、按Total revenue、按结算金额、按数量、平均到订单。
- 金额符号规则：费用填负数，返还/补贴填正数。
- 这类记录不再用于默认主链路。只有当某笔广告费既不在回款表的 `GMV Payment for TikTok Ads` 里，也不在线下广告费自动识别范围内时，才建议手工补这类规则。

当 `记录类型 = 新增调整行` 时：

- 这类行用于补原利润表中的空状态调整行，而不是去改真实订单。
- 可直接填写以下财务列：
   - Total revenue
   - Transaction fee
   - TikTok Shop commission fee
   - Seller shipping fee
   - Affiliate commission
   - Affiliate Shop Ads commission
   - Order processing fee
   - Bonus cashback service fee
   - Affiliate commission deposit
   - Shipping Service Fee
   - Voucher Xtra Service Fee
   - Pre-order service fee
   - Affiliate partner commission
   - LIVE Specials service fee
   - Campaign resource fee
   - EAMS Program service fee
   - GMV Max Coupon
   - 广告费
- 也可以直接填写覆盖列：覆盖_Total fees、覆盖_结算金额、覆盖_税费、覆盖_总计结算金额、覆盖_净利润(rmb)。
- 如果覆盖列留空，脚本会按当前程序版口径自动重算。

为什么这张表能尽量减少人工干预：

- 正常情况下，线上广告费和线下广告费都优先由系统直接生成店铺级调整行，不需要人工录入。
- 只有系统无法规则化推导的少量补贴/税费/赔付，才需要单独写成 `新增调整行`。
- 这比直接在利润表里补 543 条手工行更稳定，也更容易回查。

## 6. 原“透视”页与本次新“透视”页的关系

原利润表中的 [01-TK文件/2026年菲律宾2月tiktok订单利润表（1.1%）.xlsx](01-TK文件/2026年菲律宾2月tiktok订单利润表（1.1%29.xlsx) 的“透视”页：

- 是 28 列的三层表头
- 工作簿结构中存在 pivotCacheDefinition、pivotTable、externalLink 等对象
- 不适合直接复刻为新的 Excel 数据透视表对象

本次生成到程序版工作簿中的“透视”页：

- 不是 Excel PivotTable 对象
- 是按 [盈利分析详细精美版.xlsx](盈利分析详细精美版.xlsx) 的 52 列布局生成的静态分析页
- 冻结窗格为 C2
- 表头深蓝底白字
- 数据区使用细边框
- 利润率列使用百分比格式
- 最后三列 Order ID、Order ID.1、SKU ID 使用文本格式

## 7. 新“透视”页的生成规则

当前脚本对程序生成表里的 Tiktok订单完成表 执行以下逻辑：

1. 以 店铺 + 是否到款 作为分组键。
2. 对大部分字段先做数值化，再按分组求和。
3. 对 Order ID、Order ID.1、SKU ID 取组内首个非空值。
4. 计算字段：
   - 总收入(RMB) = Total revenue / 8.4672
   - 利润率 = 净利润(rmb) / 总收入(RMB)
5. 将结果写回 [01-TK文件/2026年菲律宾2月tiktok订单利润表_程序生成.xlsx](01-TK文件/2026年菲律宾2月tiktok订单利润表_%E7%A8%8B%E5%BA%8F%E7%94%9F%E6%88%90.xlsx) 的“透视”工作表。

关于“是否到款”字段的特别说明：

- 当前程序版订单完成表里，“是否到款 = 是” 的判断条件是：订单号能在当前 011-系统数据 对应的回款表中匹配到。
- “是否到款 = 否” 并不等于业务上认定的未到款，它只是一个技术标记，表示“未在当前脚本所扫描的回款源表中匹配到该订单号”。
- 这个标记口径与原利润表、精美版样板里的业务口径并不一致，因此不应直接拿程序版的“否”去解释业务业绩统计。

## 8. 与精美版样板不完全一致的原因

当前程序生成的“透视”页已经成功写入目标工作簿，但与 [盈利分析详细精美版.xlsx](盈利分析详细精美版.xlsx) 仍不能完全一致。

已确认原因如下。

### 8.1 分组键差异

对比结果显示，20 个分组里至少有以下 3 组状态口径不同：

- 2月份-廖楠户外：程序版技术标记为 否，样板是 是
- 2月份-阳玲品牌美妆：程序版技术标记为 否，样板是空状态
- 2月份-阳玲汽配：程序版技术标记为 否，样板是空状态

这里的“程序版为 否”不能直接解释成业务未到款，只能说明：这些订单号没有在当前 011-系统数据 的回款文件中被脚本匹配到。

进一步核对后发现：

- 程序版里被打成 否 的订单，很多在原利润表的 Tiktok到款明细表 中其实能找到。
- 示例订单 `582836580256679521` 在原利润表的 Tiktok到款明细表 里可找到对应回款记录，但在当前 011-系统数据 下的回款源文件中未找到。

据此可确认：

- 原利润表使用的回款底表，并不完全等同于当前 011-系统数据 目录里这批回款源文件。
- 程序版把这部分订单标成 否，根因是“当前源数据缺少对应回款匹配”，而不是“这些订单业务上未到款”。
- 因此前一版文档把这些分组直接表述为“到款情况为否”，语义过重，这里更正为“技术匹配结果为否”。

### 8.2 金额差异来源于原利润表与当前系统重建结果不一致

示例：

- 分组：2月份-刘林长3C / 是
- 程序版透视 Total revenue = 1298025.05
- 精美版样板 Total revenue = 1140350.61
- 程序版透视 广告费 = 0
- 精美版样板 广告费 = -3549.42

这说明精美版样板使用的底层订单完成表，与当前依据 011-系统数据 重建出的程序版订单完成表并不相同。

### 8.3 广告费未重建导致差异持续存在

当前对比中，广告费列的最大差值达到 81955.76。其根因不是透视算法，而是底层订单完成表缺少原利润表中的人工广告费调整行。

### 8.4 原利润表费用列历史错位会继续影响样板结果

由于精美版样板显然是基于原利润表订单完成表继续聚合出来的，所以它会继承原利润表里那套历史错位的费用列口径。

当前程序版透视页按“源表真实字段语义”聚合，因此在费用分类上会和样板出现系统性差异。

## 10. 下一步建议

如果要继续逼近原精美版结果，优先顺序建议如下：

1. 先导出并填写 [01-TK文件/广告费导入表.xlsx](01-TK文件/%E5%B9%BF%E5%91%8A%E8%B4%B9%E5%AF%BC%E5%85%A5%E8%A1%A8.xlsx)，优先每店每月只写 1 条广告费分摊规则。
2. 再把确实无法从系统规则推出的 Withholding tax、Platform reimbursement、Logistics reimbursement 等写成 `新增调整行`。
3. 最后再决定是否要兼容原利润表那套历史错位列口径；如果目标是“真实口径正确”，应继续沿用当前程序版字段语义，仅把精美版当样式模板。
